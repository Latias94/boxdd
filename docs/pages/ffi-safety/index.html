FFI Safety

The safe boxdd crate wraps Box2D's C API with explicit ownership, callback, raw pointer, and event-buffer rules. This page points to the audit trail and the tests that keep those rules executable.

Principles

Single-owner world

World, owned handles, and the Bevy physics context stay on one thread. Worker-count configuration does not make the public world API sendable.

Callback containment

Rust closures invoked by Box2D are guarded so panics are caught before crossing C callbacks and resumed after Box2D returns to Rust.

Explicit raw seams

Raw ids, raw event slices, raw transforms, and raw user-data pointers use named *_raw, from_raw, or into_raw APIs so FFI ownership stays visible.

Hazards And Evidence

Hazard Guard Evidence
Calling Box2D while it has locked the world for a callback. Callback guard returns ApiError::InCallback on try_* paths and panics on misuse-style convenience paths. try_api.rs, panic_across_ffi_is_caught.rs
Rust panic escaping through a C callback boundary. Callback trampolines use panic capture and resume unwinding after the native call completes. panic_across_ffi_is_caught.rs, dynamic_tree.rs
Borrowing Box2D event buffers after the next step. Owned snapshots copy data; view/raw access is closure-scoped and defers destroys until the borrowed scope exits. events_and_sensors.rs, user_data.rs
Stale ids after body, shape, joint, or chain destruction. Safe handle methods validate ids; recoverable paths return ApiError::Invalid*Id instead of entering native code with stale ids. handle_validity_panics.rs, world_destroy_and_recycle.rs, ffi_lifecycle.rs
Caller-owned raw user-data pointers outliving Rust values. Raw pointer setters stay unsafe and explicitly named; typed user data is owned by the world store and dropped on clear/destroy paths. user_data.rs, ffi_lifecycle.rs
Public world and handle types accidentally becoming sendable. Public world, owned handle, and Bevy context types are compile-asserted as !Send/!Sync. ffi_lifecycle.rs, plugin.rs

When Adding Wrappers

Before exposing a safe convenience API, check whether the upstream function borrows caller memory after return, invokes a Rust closure from C, returns pointers into Box2D-owned storage, can run during callbacks, or destroys objects that invalidate ids stored elsewhere. If any answer is yes, update the lifetime audit and add focused tests before release.
